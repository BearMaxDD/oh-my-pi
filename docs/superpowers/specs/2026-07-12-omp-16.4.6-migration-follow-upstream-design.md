## 1. 问题陈述

当前个人增强版 OMP 存在三层结构性问题：

1. **基线落后**：自研挂在 16.3.3；目标核心已是 16.4.6，且 16.4.6 目录不是 git 仓库，无法直接 3-way merge。
2. **自研与核心平铺混写**：PlanRun、严格角色路由、Superpowers、codebase-memory、Plan Gate、分段写等约 250 个变更文件几乎全部落在 `packages/coding-agent`，其中大量是核心路径修改（task/session/config/modes）。
3. **跟版成本高**：每次升 upstream 时冲突分散在实现文件与接线文件之间；已有“独立仓 omp-custom”PRD，但仓库与抽离尚未落地。

目标不是“把 16.3.3 补丁硬贴到 16.4.6”，而是：

- 以 **16.4.6 为唯一新核心**；
- 把自研主体迁到 **独立仓库**；
- fork 只保留 **可枚举的最小接线补丁**；
- 建立以后升 16.x.y 的 **可复制 runbook**。

## 4. 目标架构

### 4.1 仓库拓扑

```text
can1357/oh-my-pi @ v16.4.6          # 官方上游（只读跟随）
        │ merge tags
        ▼
BearMaxDD/oh-my-pi                  # 新主线：16.4.6 基线 + 接线白名单
  - 官方 monorepo 原样保留
  - 仅白名单文件可含自研接线
  - 依赖 @bearmaxdd/omp-custom
        │ package / git / path dependency
        ▼
BearMaxDD/omp-custom                # 自研主体（新建）
  - plan-run / routing / superpowers
  - codebase-memory / plan-gate / tools
  - host/omp 适配与注册 API
  - 独立 test / check / version
        │
        └── adapters/（后置）         # Codex 等第二宿主
```

### 4.2 硬约束

1. **16.4.6 是唯一新核心**；不以 16.3.3 继续堆功能。
2. **自研实现不得平铺混写**进 `packages/coding-agent/src/**`（接线白名单文件除外）。
3. **核心补丁必须可枚举**：白名单文档化；白名单外改核心视为设计违规。
4. **依赖方向单向**：
   - `oh-my-pi` → `@bearmaxdd/omp-custom`
   - `omp-custom` 可依赖 `@oh-my-pi/*` 的**公开类型/SDK/窄接口**（优先 types-only）
   - 禁止 `omp-custom` 依赖 `oh-my-pi` 内部相对路径
   - 禁止 coding-agent 核心 import omp-custom 深层实现（只走 host API）
5. **禁用扩展时核心仍可启动**。

## 5. 组件设计

### 5.1 `@bearmaxdd/omp-custom` 模块

| 模块 | 职责 | 主要来源（16.3.3 自定义树） |
| --- | --- | --- |
| `src/plan-run/**` | PlanRun 状态机、driver、execution-book、repair、evidence、scheduler、segmented-write | `packages/coding-agent/src/codex-plan-run/**` |
| `src/routing/**` | 严格角色绑定、role contract、批量分配服务、审计、routing evidence schema | `task/strict-*`、`config/model-role-*`、`config/role-model-audit.ts`、`modes/components/model-role-bulk-assignment.ts` 等 |
| `src/superpowers/**` | agent-bridge、技能/codebase-memory 门禁 | `src/superpowers/**` |
| `src/codebase-memory/**` | autocontext、recon、reindex、execution-gate | `codebase-memory-autocontext.ts`、plan-run 内 recon/reindex/gate |
| `src/plan-gate/**` | 执行书门禁、smoke | `src/plan-gate/**` |
| `src/tools/**` | plan-execution-book、plan-repair-loop、code-write-policy 等实现 | `src/tools/plan-*.ts`、`code-write-policy.ts` |
| `src/commands/**` | 命令 handler 实现（非官方 registry 本体） | `commands/plan-run.ts`、`commands/superpowers.ts` 等逻辑 |
| `src/host/omp/**` | 对 OMP 的稳定注册/接线 API：`createOmpCustomExtension`、`registerOmpCustom`、钩子回调类型 | 新建 |
| `src/extension.ts` | `omp.extensions` 入口（registerCommand/Tool/on） | 新建 |
| `src/index.ts` | 稳定公共导出 | 新建 |
| `test/**` | 按模块镜像迁移的单测 | `packages/coding-agent/test/**` 对应子集 |

### 5.2 `BearMaxDD/oh-my-pi` 接线白名单（目标形态）

以下为**目标白名单类别**（实现计划阶段落到具体路径；合计目标 ≤10 个核心路径，另加 package 元数据）：

1. **依赖声明**：root/`coding-agent` package.json（及必要 catalog）引入 `@bearmaxdd/omp-custom`
2. **扩展加载**：extension / bundled registration 入口
3. **命令注册**：`/plan-run`、`/superpowers`、`models roles --check` 的 registry 挂钩
4. **工具注册**：plan tools / code-write-policy 的 builtin 或 extension 注册点
5. **settings schema/读写钩子**：`superpowers.*`、严格路由相关字段；批量原子写入若需 settings 扩展则单点适配
6. **spawn 前断言钩子**：`task/executor`（或等价官方 spawn 入口）最小调用点
7. **`/model` UI 入口**：批量分配菜单入口，调用 omp-custom 状态机（不把状态机实现留在 fork）
8. **session modelLock 钩子**：仅当公开 API 不足以保证严格语义时启用；feature flag 保护

**白名单外的 coding-agent 核心文件默认禁止修改。**  
若实现中发现必须新增白名单项：先更新本规格/接线清单，再改代码。

### 5.3 与 16.4.6 扩展面的映射

- **优先**：`ExtensionAPI`（`registerCommand` / `registerTool` / `on(event)`）承载命令、工具、autocontext。
- **严格路由**：官方 extension 无法保证 spawn 前不变量时，使用白名单 spawn/session 钩子。
- **model-resolver**：保留官方宽松语义；严格模式是 omp-custom 上层封装，**不修改 resolver 本体**作为默认策略。
- **16.4.6 新增能力**（vibe、model-hub、search providers 等）默认不动；不与自研抢入口，除非验收发现冲突再单列适配。

## 6. 数据流

### 6.1 `/model` 批量分配

```text
UI 入口(fork 白名单)
  → routing bulk-assignment reducer/service (omp-custom)
  → settings 原子写入钩子 (fork)
  → role-model audit 可 `models roles --check`
```

### 6.2 PlanRun / Role-bound Task

```text
命令或任务入口 (fork 注册)
  → plan-run driver (omp-custom)
  → role contract + strict binding (omp-custom)
  → spawn 前断言钩子 (fork 白名单)
  → 官方 task/subagent 执行
  → evidence 写入 (omp-custom)
  → 状态面板/验收消费
```

### 6.3 普通代码对话 + Superpowers

```text
input/context 事件 (Extension)
  → codebase-memory autocontext (omp-custom)
  → 可选 Superpowers 门禁 (omp-custom)
  → 工具调用与 evidence
```

## 7. 错误处理

| 场景 | 行为 |
| --- | --- |
| 严格路由缺绑定 / 契约失败 / 实际模型不一致 | **硬失败**，给出可修复提示；禁止静默回退会话模型 |
| PlanRun 门禁失败 | 进入 repair/证据路径；不得假装通过 |
| omp-custom 扩展加载失败 | 核心 CLI 仍可启动；自研命令/工具不可用并有明确提示 |
| codebase-memory MCP 不可用 | 按 settings：`off` / `advisory` / `required` 降级或阻断 |
| 跟版后白名单外出现自研 diff | 视为违规：回退或迁回 omp-custom，禁止“先合再说” |
| spawn 钩子因上游 API 变化失效 | feature flag 降级 + 明确错误；优先修 host 适配层 |

## 8. 一次性迁移流程

1. **基线化**
   - 将 `oh-my-pi-16.4.6` 初始化为 git 仓库（或导入为新主线工作树）。
   - 对齐 upstream `v16.4.6` tag/提交语义；配置 `upstream`/`origin`。
2. **抽包脚手架**
   - 创建 `BearMaxDD/omp-custom`：package.json、check/test 脚本、`src/index.ts`、`src/extension.ts`、`src/host/omp/**`。
3. **按耦合度迁移实现**
   - P1 低耦合：codebase-memory、plan-gate、segmented-write
   - P2 路由：strict routing、bulk assignment、审计 + 白名单钩子
   - P3 PlanRun + Superpowers + tools/commands
4. **接线**
   - 仅改白名单；接通依赖与注册。
5. **回归**
   - 对照 16.3.3 自定义版用户可见行为清单验收。
6. **切主**
   - 新主线设为默认开发入口；旧 16.3.3 树只读参考。

## 9. 跟版 runbook（每次升 16.x.y）

标准步骤：

1. `git fetch upstream --tags`
2. merge/rebase `vX.Y.Z` 到 fork main
3. **只检查接线白名单冲突**
4. 若官方 API 变化：改 `omp-custom/src/host/omp/**` 适配层（优先），必要时修订白名单并更新文档
5. 跑 `omp-custom` 单测 + fork 冒烟（扩展加载、关键命令、严格路由硬失败样例、PlanRun 入口）
6. 更新 `FOLLOW_UPSTREAM.md`（或等价文档）变更记录：版本、冲突文件、适配点、验证命令与结果

成功判据（可测量）：

- 核心补丁路径有白名单文档，默认 ≤10
- 合入新 tag 后，预期冲突主要落在白名单
- `omp-custom` 可独立 `bun test` 通过
- runbook 命令序列可复制执行

## 12. 风险与缓解

| 风险 | 影响 | 缓解 |
| --- | --- | --- |
| 16.4.6 相对 16.3.3 核心 API 漂移（task/session/modes） | 迁移编译/行为失败 | host 适配层集中消化；先契约测试后接线 |
| 严格路由所需钩子超过预算 | 白名单膨胀 | 优先扩展 API；每增一项必须文档变更+理由 |
| PlanRun 体量大 | 工期与回归成本高 | 模块化迁移 + 证据/状态机单测先行 |
| 一次切主后旧环境依赖 16.3.3 路径 | 使用中断 | 明确默认入口与只读参考策略；必要时短期 wrapper 脚本 |
| 16.4.6 快照无历史 | 难 3-way | 基线化时对齐 upstream tag，建立真正 merge 祖先 |
